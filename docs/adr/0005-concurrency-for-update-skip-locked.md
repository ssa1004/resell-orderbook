# ADR-0005: 매칭 동시성 — Postgres advisory lock + FOR UPDATE SKIP LOCKED

## 상태
적용

## 배경
매칭은 *읽고 → 판단하고 → 쓰는 (read-modify-write)* 흐름을 하나의 단위로 처리해야 한다.

1. 같은 SKU 의 best 호가 (가장 높은 BID 또는 가장 낮은 ASK) 를 select.
2. 가격 비교.
3. 매칭이면 두 호가의 status 를 MATCHED 로 변경 + Trade INSERT.

여러 트랜잭션이 동시에 같은 SKU 에 도달하면, *같은 best 호가* 를 두 번 매칭할 수 있다 →
데이터 일관성이 깨진다.

## 결정
두 종류의 잠금을 같이 쓴다.

1. **`pg_advisory_xact_lock(hash(sku_id))`** — PostgreSQL 의 응용 락 (테이블이나 행이 아니라
   임의의 정수/해시 값에 잠금을 걸 수 있는 락). 같은 SKU 의 매칭 시도를 SKU 단위로 한 줄로
   줄세운다. 트랜잭션이 끝나면 자동 해제. 같은 키로만 락을 잡으니 *서로 다른 두 트랜잭션이
   서로의 락을 기다려서 멈추는* 데드락이 구조적으로 발생할 수 없다.
2. **`SELECT ... FOR UPDATE SKIP LOCKED`** — best 호가 한 행을 행 단위 비관적 락으로 잡는다.
   - `FOR UPDATE` 는 그 행을 트랜잭션 끝까지 다른 트랜잭션이 변경할 수 없도록 잠근다.
   - `SKIP LOCKED` 는 다른 트랜잭션이 이미 잠근 행을 *대기 없이 건너뛴다*. 이게 없으면
     동시에 같은 SKU 를 노리는 두 트랜잭션이 서로를 기다리며 직렬 처리 — 처리량이 떨어진다.
     건너뛰면 두 번째 트랜잭션이 그 다음 best 호가를 즉시 잡아 매칭 시도.
   - 결과: 다른 SKU 의 매칭과 섞이지 않으면서 같은 SKU 안에서도 큐가 빠르게 흐른다.

분산 락 (여러 서버 인스턴스 사이에서 공유하는 잠금. 보통 Redis 의 Redisson 같은 라이브러리로
구현) 은 지금은 도입하지 않는다. DB 트랜잭션과 자연스럽게 묶이는 advisory lock 이 더 단순하다.

## 장단점
- 같은 SKU 의 매칭이 한 줄로 줄세워져 정합성이 보장되고, 다른 SKU 는 병렬이라 처리량은
  보존된다.
- DB 외부 의존이 없다 — 한 트랜잭션 안에서 락 획득과 해제가 모두 끝난다.
- Postgres 전용 기능이라 H2 dev 환경에서는 `market.advisory-lock.enabled=false` 로 끈다
  (개발 환경은 인스턴스 한 대라 동시 매칭 경쟁이 사실상 없음).
- 인기 SKU 매칭 빈도가 초당 100건 넘어가는 시점에는 DB 락 경합이 생길 수 있다 — 그 때
  Redis 분산 락 + 호가 캐싱 도입을 검토한다.

## 다른 선택지를 안 쓴 이유
- **분산 락 (Redisson) 만 쓰기**: DB 트랜잭션과 별개로 동작해서 잠금을 언제 풀지, 풀리지
  않고 누수되는 경우 (lock leak) 등을 직접 신경 써야 한다. 더 복잡하다.
- **행 단위 비관적 락만 (advisory 없이)**: 두 SKU 의 호가를 락 잡는 순서가 트랜잭션마다
  달라질 수 있어 데드락이 가능하다. 코드 작성 패턴에 의존하게 되고 쉽게 깨진다.
