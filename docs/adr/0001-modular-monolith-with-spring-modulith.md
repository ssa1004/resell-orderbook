# ADR-0001: Spring Modulith 기반 모듈러 모놀리스

## 상태
적용

## 배경
한정판 리셀 마켓의 거래 흐름은 catalog → trading → inspection → settlement 라는 단일 도메인 안에서 강하게 묶여 있다. 트랜잭션, 일관성, 응답 지연 모두 한 프로세스 안에서 풀어야 자연스럽다.

다만 모놀리스로 가더라도 모듈 경계 자체는 명확히 강제되어야 한다. 관습이나 코드 리뷰만으로는 시간이 지나면서 경계가 흐려진다.

## 결정
**Spring Modulith (Spring 의 모듈 경계 검증 라이브러리) 로 모듈 경계를 빌드 시점에 검증하는
모듈러 모놀리스 (한 배포 단위 안에서 모듈 경계를 강하게 지키는 구조) 로 간다.**

각 하위 패키지에 `@NamedInterface` 로 외부에 노출할 타입을 명시하고,
`ApplicationModules.verify()` 가 빌드 단계에서 의존성 위반을 잡아낸다.

## 장단점
- 한 프로세스 / 한 트랜잭션이라 Outbox + 매칭 + 정산을 자연스럽게 묶을 수 있다.
- MSA (마이크로서비스 아키텍처) 의 운영 복잡도 (서비스 메쉬, 분산 트레이싱, 분산 트랜잭션)
  를 피한다.
- 모듈 경계 위반이 빌드 단계에서 잡힌다 (테스트로 늦게 발견되는 게 아니라 컴파일/검증
  단계에서 차단).
- 한 인스턴스가 죽으면 전체가 함께 멈춘다 — k8s 의 복제 (replicas) + readiness probe
  (트래픽 받을 준비가 됐는지 검사하는 헬스체크) 로 보완.
- 나중에 MSA 로 분리하더라도 모듈 경계가 지켜져 있으면 비용이 크지 않다.

## 다시 검토할 시점
- 거래량이 인스턴스 한 대로 처리 불가능한 수준 (피크 초당 1만 매칭+) 에 도달했을 때
- 검수 도메인이 별도 운영 조직으로 분리되어 릴리스 주기가 어긋날 때
