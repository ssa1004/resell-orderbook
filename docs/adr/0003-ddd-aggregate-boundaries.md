# ADR-0003: Listing/Bid/Trade 를 각각 별도 애그리거트로

## 상태
적용

## 배경
호가창은 같은 SKU 의 ASK (판매 호가) 들 + BID (구매 호가) 들 + 체결된 Trade 들의 묶음이다.
이걸 도메인 모델로 표현하는 두 가지 후보가 있다.

- **A**: `OrderBook` 을 애그리거트 루트 (DDD 에서 한 트랜잭션이 잠금 단위로 다루는 객체
  덩어리의 진입점) 로 두고 모든 ASK/BID/Trade 를 그 안에 둔다.
- **B**: `Listing`, `Bid`, `Trade` 를 각각 별도 애그리거트로 둔다.

## 결정
**B 안 — 분리한다.**

OrderBook 은 실제 저장되는 객체가 아니라 조회로 만들어내는 결과물로 본다.
`OrderBookQueryUseCase` 라는 읽기 전용 인터페이스로 표현한다.

## 장단점
- 인기 SKU (예: 인기 Yeezy) 의 모든 호가가 한 락에 잡혀버리는 경쟁을 피한다.
- 애그리거트가 작아 DB 에서 읽고 쓰는 비용이 낮다.
- 매칭 시 best Bid (가장 높은 매수 호가) / best Ask (가장 낮은 매도 호가) 한 행만 락
  잡으면 직렬화에 충분하다.
- "호가창 전체의 불변식 (항상 지켜져야 하는 규칙)" 이 도메인 한 곳에 모이지는 않음 —
  호출자(`PlaceListingService`) 가 조정한다.
  - 보완: 모든 매칭 service 가 동일한 흐름 패턴을 따르므로 코드 리뷰에서 일관성 확인 가능.

## A 안을 선택했을 때의 문제
인기 SKU 1개의 매칭에 호가창 전체가 락에 걸린다. 분당 수만 건 매칭이 발생하는 hot SKU
(거래가 몰리는 인기 상품) 에서 처리량이 떨어진다.
